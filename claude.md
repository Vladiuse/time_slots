# Техническое задание: система записи на выгрузку контейнеров

## 1. Контекст

Железнодорожная контейнерная станция выдаёт клиентам контейнеры. Клиенты приезжают на фурах и забирают свои контейнеры. Сейчас выдача идёт по живой очереди: фура приехала — ждёт, пока не разгрузят все фуры перед ней.

## 2. Цель

Автоматизировать процесс выдачи контейнеров: внедрить систему предварительной записи клиентов на конкретный временной слот, чтобы сократить время ожидания на станции.

## 3. Глоссарий

| Термин          | Определение                                                                                  |
|-----------------|----------------------------------------------------------------------------------------------|
| Станция         | Железнодорожная контейнерная станция, на которой хранятся и выдаются контейнеры              |
| Контейнер       | Груз, принадлежащий клиенту, ожидающий выгрузки                                              |
| Слот            | Фиксированный интервал времени, в течение которого клиент может забрать контейнеры           |
| Заявка          | Запись клиента на конкретный слот с привязкой к одному или нескольким контейнерам            |
| Клиент          | Юридическое лицо — владелец контейнеров                                                      |
| Администрация   | Сотрудники станции, управляющие заявками                                                     |

## 5. Бизнес-правила слотов

- Слоты создаются **ежедневно**.
- В сутки **6 слотов по 4 часа** (24 часа покрыты полностью).
- Лимит на один слот — **50 контейнеров**.
- Заявка может содержать **один или несколько** контейнеров.
- Если суммарное количество контейнеров в заявках на слот достигло 50 — слот считается занятым и недоступен для новых заявок.
- Клиент может **отменить свою заявку**, только если время начала слота ещё в будущем.
- Администрация может **закрыть приём заявок** для всех клиентов (например, при ЧП).

## 6. Функциональные требования

### 6.1. Интерфейс клиента
- Просмотр **своих** контейнеров на станции (других клиентов не видит).
- Создание заявки: выбор контейнеров + выбор доступного слота.
- Просмотр активных заявок (на будущие слоты).
- Отмена своей активной заявки (если слот ещё не наступил).
- Просмотр истории: выполненные заявки и уже выгруженные контейнеры.

### 6.2. Интерфейс администрации
- Просмотр всех заявок (по дням, слотам, клиентам).
- Отмена заявки клиента с обязательным **уведомлением клиента** об отмене.
  - TBD: канал уведомления (email / SMS / in-app).
- Закрытие приёма заявок для всех клиентов.
- TBD: управление аккаунтами клиентов (создание, блокировка) — нужно ли в MVP.

## 7. Источник данных

### 7.1. Контейнеры
- Источник — внешняя БД (FoxPro), доступная только через выгрузку в текстовый файл.
- В файле для каждого контейнера: **номер**, **дата прибытия**, **принадлежность клиенту**.
- Прямой доступ к FoxPro отложен на будущее.

### 7.2. Клиенты
- Таблица клиентов ведётся вручную внутри системы (не выгружается из FoxPro).

### 7.3. Синхронизация
- Каждый час система получает новый текстовый файл.
- Алгоритм: вычислить разницу между состоянием БД и файлом, применить **только изменения** (добавить новые контейнеры, обновить изменённые, удалить отсутствующие).
